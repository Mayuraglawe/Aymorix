## 12. Assignment and Proctoring System

### 12.1 Assignment System

The Platform provides tools for creating, distributing, and grading assignments. Faculty are responsible for:
- Ensuring the accuracy and appropriateness of assignment content.
- Setting fair deadlines and grading criteria.
- Reviewing auto-graded results for accuracy.

### 12.2 Proctoring Features

The Platform includes browser-based proctoring features that monitor the following during assessments:
- **Tab Switching**: Detection of navigation away from the assessment tab.
- **Window Blur/Focus**: Detection of browser window losing focus.
- **Copy-Paste Activity**: Detection of clipboard usage.
- **Right-Click Actions**: Detection of context menu usage.
- **Developer Tools**: Detection of browser developer tools opening.
- **Fullscreen Monitoring**: Detection of exiting fullscreen mode.

### 12.3 Proctoring Limitations and Acknowledgment

**Students acknowledge that:**
- Proctoring is **browser-based** and operates within the technical limitations of web browsers.
- Proctoring does **not** include webcam monitoring, screen recording, or audio surveillance.
- The system may generate **false positives** (flagging legitimate behavior as violations) due to accidental tab switches, system notifications, browser updates, or other benign events.
- Proctoring logs are meant to **assist** Faculty and Administrators in evaluating academic integrity; they are **not** definitive proof of misconduct.
- Flagged violations should be reviewed in context by Faculty before any disciplinary action is taken.

**Faculty and Administrators acknowledge that:**
- Proctoring data should be interpreted carefully and in conjunction with other evidence.
- Students should be informed about the types of monitoring active during assessments.
- Proctoring is a **supplementary** tool and should not be the sole basis for academic integrity decisions.

### 12.4 Student Submissions

- All assignment submissions (text, code, essay responses) are stored on the Platform.
- Students retain intellectual ownership of their original submissions.
- The Institution and Faculty have the right to review, grade, and store submissions for academic evaluation purposes.
- Submissions may be retained for academic record-keeping in accordance with institutional policies.

## 17. Data Security and Multi-Tenancy

### 17.1 Multi-Tenant Architecture

Academic Compass operates a **multi-tenant architecture** where multiple Institutions share the same Platform infrastructure while maintaining strict data isolation. Key security measures include:

- **Row-Level Security (RLS)**: Database-level policies ensure that users can only access data belonging to their Institution and department.
- **Department-Based Isolation**: Faculty can only access data within their assigned department. Students see only data relevant to their batch and courses.
- **Role-Based Access Control (RBAC)**: Permissions are granted based on user roles, with the principle of least privilege applied.
- **Institutional Boundaries**: Data from one Institution is never accessible to users of another Institution.

### 17.2 Security Measures

We implement the following security measures:
- Encrypted data transmission using HTTPS/TLS.
- Secure password hashing (passwords are never stored in plain text).
- Token-based authentication with session expiration.
- Rate limiting to prevent brute force attacks.
- Input validation and sanitization to prevent injection attacks.
- Regular security assessments and updates.

### 17.3 Security Incident Response

In the event of a data breach or security incident:
- We will investigate the incident promptly and take appropriate remedial action.
- Affected Institutions and Users will be notified in accordance with applicable data protection laws and regulations.
- We will cooperate with relevant authorities as required.

### 17.4 User Security Obligations
